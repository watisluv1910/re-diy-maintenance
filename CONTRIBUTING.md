
# Руководство для контрибьюторов

Этот документ описывает процесс внесения изменений в репозиторий.

**!** В данном проекте используется и должен соблюдаться [Google Markdown Style Guide](https://google.github.io/styleguide/docguide/style.html).

## Как помочь проекту

Вы можете:

- Исправить опечатки или ошибки в существующих руководствах.
- Добавить новое руководство/пошаговую инструкцию по обслуживанию мотоциклов Royal Enfield.
- Улучшить оформление или переработать структуру документов.
- Предложить идеи по улучшению проекта в [issues](https://github.com/watisluv1910/re-diy-maintenance/issues).

## Процесс внесения изменений

### 1. Создайте Issue

Любое изменение должно быть связано с открытой *issue*. Это позволяет обсудить задачу и избежать дублирования работы:
- Проверьте, нет ли уже похожей *issue*.
- Если нет — создайте новую, чётко описав, что вы предлагаете изменить или добавить.

### 2. Создайте ветку

Ветку необходимо создавать от актуальной версии `main`.  
Имя ветки должно строго соответствовать шаблону:

```
<номер-issue>_<username>_<предмет-изменений>
```

Пример: `15_nasevich_meteor-chain-adjustment-guide`

### 3. Внесите изменения

#### Общие правила

- Все изменения должны касаться только файлов в директории `content/`, т.е. содержимого базы знаний.
- Если вы хотите предложить правки в конфигурации Quartz или других системных файлах, создайте *issue* для обсуждения с меткой `config`.
- Следите за тем, чтобы форматирование markdown-файлов было единообразным с остальными документами в репозитории.

#### Коммиты

Сообщение каждого коммита должо следовать спецификации [Conventional Commits](https://www.conventionalcommits.org/).

Основные типы:
- `feat` – новое руководство или крупная функциональность.
- `fix` – исправление ошибки.
- `docs` – изменения в документации (включая этот файл).
- `style` – правки форматирования, не меняющие смысла.
- `refactor` – переработка существующего текста без изменения содержания.

В коммитах допускаются сообщения как на русском, так и на английском языках. Пример:

```
feat: добавить инструкцию по замене тормозной жидкости для Meteor 350
```

### 4. Создайте Pull Request

- Откройте *PR* из вашей ветки в `main`.
- В заголовке и описании PR обязательно укажите номер *issue*.
- Кратко опишите, что было сделано.
- Убедитесь, что локальная сборка (`npx quartz build`) проходит без ошибок, и сайт отображается корректно.
- *PR* не должен содержать изменения за пределами директории `content/`, если это не было согласовано в отдельной issue.

### 5. Ревью и слияние

- *PR* будет рассмотрен владельцем репозитория или одним из коллабораторов.
- Если потребуются доработки, вы получите замечания. Вносите правки в ту же ветку и обновляйте *PR*, **не создавайте новый**.
- После одобрения и успешной проверки PR будет слит с сохранением истории коммитов.

## Важные правила

- Не вносите изменения напрямую в ветку `main`.
- Не включайте в *PR* файлы, не относящиеся к вашим правкам (например, временные файлы редактора, папку `public/`, `node_modules/`).
- Не забывайте указывать ссылки на источники информации в отдельной секции в конце заметки.
- Уважайте чужой труд и будьте конструктивны в обсуждениях.

## Вопросы?

Если что-то осталось непонятным, создайте *issue* с вопросом или свяжитесь с владельцем репозитория.
